# Internationalisation (i18n)

Django ships the i18n machinery. The bits a starter usually misses: `LOCALE_PATHS`, `LANGUAGES`, `LocaleMiddleware`, the locale-aware URL include, and a `makemessages` / `compilemessages` workflow.

Skip if the project will only ever speak one language. Adding it later costs real work — every existing template string needs `{% trans %}` and every Python user-facing string needs `gettext_lazy`.

## Settings

```python
from django.utils.translation import gettext_lazy as _

LANGUAGE_CODE = "en"
USE_I18N = True
USE_TZ = True

LANGUAGES = [
    ("en", _("English")),
    ("de", _("German")),     # add the languages you intend to ship
]

LOCALE_PATHS = [BASE_DIR / "locale"]
```

`USE_L10N` is gone in Django 5+ — formats follow `LANGUAGE_CODE` automatically.

## Middleware

`LocaleMiddleware` reads the user's language from the `Accept-Language` header / session / cookie / URL prefix, in that order. Insert immediately before `CommonMiddleware`:

```python
common_idx = MIDDLEWARE.index("django.middleware.common.CommonMiddleware")
MIDDLEWARE.insert(common_idx, "django.middleware.locale.LocaleMiddleware")
```

## URLs — language prefix (optional)

Pick **one** of two patterns:

**A. No prefix.** Language is detected from header / cookie. URLs stay `/admin/`, `/foo/`. Simplest.

**B. Language prefix.** URLs become `/en/admin/`, `/de/admin/`. Better for SEO, link-shareable.

```python
from django.conf.urls.i18n import i18n_patterns
from django.urls import include, path

urlpatterns = [
    path("i18n/", include("django.conf.urls.i18n")),   # the set_language POST view
    *i18n_patterns(
        path("admin/", admin.site.urls),
        # other prefixed patterns here
        prefix_default_language=False,
    ),
]
```

`prefix_default_language=False` keeps the default-language URLs unprefixed (`/admin/` instead of `/en/admin/`).

## Templates

```django
{% load i18n %}
<title>{% trans "Sign in" %}</title>
<p>{% blocktrans with email=user.email %}Hello, {{ email }}{% endblocktrans %}</p>
```

Switch language form (works because of `i18n/` URL include above):

```django
<form action="{% url 'set_language' %}" method="post">
  {% csrf_token %}
  <select name="language" onchange="this.form.submit()">
    {% get_current_language as cur %}
    {% get_available_languages as langs %}
    {% for code, name in langs %}
      <option value="{{ code }}" {% if code == cur %}selected{% endif %}>{{ name }}</option>
    {% endfor %}
  </select>
</form>
```

## Workflow

Generate / update `.po` files for each language:

```sh
uv run manage.py makemessages -l de -l fr
```

Edit `locale/<lang>/LC_MESSAGES/django.po`. Compile:

```sh
uv run manage.py compilemessages
```

Commit `.po` files. Don't commit `.mo` files (compile at deploy / Docker build time):

```
# add to .gitignore
*.mo
```

In production Dockerfile, after `COPY . .`:

```dockerfile
RUN python manage.py compilemessages
```

## Pragmatics

- `gettext_lazy` (alias `_`) for module-level strings — they're evaluated at import, before any request, so eager `gettext` would freeze them in the server's default language.
- `gettext` (alias `_` — note: same name; use one or the other consistently) inside view bodies and request-scoped code.
- JavaScript translations: `JavaScriptCatalog` view. Skip until you actually need translated client-side strings.
